MCP Gateway

Middleware as Configuration

Evolving mcp-proxy into an intelligent composition layer

Fork of sparfenyuk/mcp-proxy

The Problem

Raw MCP Integration

Context Window Bloat

  • Each tool exposes all parameters
  • Complex output schemas consume tokens
  • Irrelevant fields create noise
  • No control over what LLM sees
{
  "entities": [
    {"name": "Alice", "entityType": "person",
     "observations": ["...", "...", "..."],
     "metadata": {...}, "timestamps": {...}}
  ]
}
// When you only needed: ["Alice"]

The Analogy

"If LLMs are an Operating System and MCP servers are libraries,
raw MCP integration is like unrolling library source into main()"

We need a composition layer

The Solution

MCP Gateway

Gateway Architecture


Agent/LLM  ──►  MCP Gateway  ──►  Backend MCP Servers
              │
              ├─ Tool Adaptation
              ├─ Schema Projection
              ├─ Default Injection
              └─ Output Transformation

All transformation is declarative configuration

MVA1: Tool Adaptation Layer

src/mcp_proxy/config_loader.py

Feature: Tool Renaming

Expose backend tools under different names via source field

{
  "tools": [
    {
      "name": "research_url",
      "source": "fetch",
      "description": "Fetch and analyze a web page",
      "server": {"command": "uvx", "args": ["mcp-server-fetch"]}
    }
  ]
}

LLM sees research_url, gateway calls fetch

Feature: Field Defaults & Hiding

Inject values and remove from schema

{
  "name": "get_nyc_weather",
  "source": "get_weather",
  "defaults": {
    "city": "New York",
    "api_key": "${WEATHER_API_KEY}"
  },
  "hide_fields": ["units", "format"]
}
  • Fields with defaults are auto-hidden
  • Environment variables supported
  • LLM never sees hidden params

Feature: Output Schema Projection

JSONPath expressions to transform responses

src/mcp_proxy/output_transformer.py

{
  "name": "list_entity_names",
  "source": "read_graph",
  "outputSchema": {
    "type": "object",
    "properties": {
      "names": {
        "type": "array",
        "items": {"type": "string"},
        "source_field": "$.entities[*].name"
      }
    }
  }
}

Result: {"names": ["Alice", "Bob", "Carol"]}

JSONPath Support

PatternExampleResult
Root path$.foo.barNested value
Array index$.items[0].nameFirst element
Wildcard$.items[*].idAll IDs as array
Deep nested$.a.b.c.dDeeply nested value

Phase 1: JSON-in-Text

Many MCP servers return JSON inside text fields

// What servers return:
{
  "content": [{
    "type": "text",
    "text": "Here is the data: {\"temp\": 72.5, \"humidity\": 45}"
  }]
}

// What we want:
{
  "structuredContent": {"temp": 72.5, "humidity": 45}
}

JSON Detection Strategies

  1. Pure JSON parsing - Direct parse attempt
  2. Pattern-based extraction - Find JSON after text prefixes
  3. Balanced brace extraction - Handle trailing text
  4. Multiple fallbacks - Graceful degradation
# Examples handled:
'{"foo": "bar"}'                    # Pure JSON
'Response: {"result": 42}'          # Prefixed
'Data: {"a": 1} more text'          # Trailing
'{"nested": {"deep": "value"}}'     # Complex

Phase 2: Markdown List Parsing

For servers returning formatted text (search, issues, etc.)

{
  "name": "search_results",
  "textExtraction": {
    "parser": "markdown_numbered_list",
    "list_field": "results",
    "item_patterns": {
      "title": {"regex": "\\*\\*([^*]+)\\*\\*", "required": true},
      "url": {"regex": "https?://[^\\s]+"},
      "stars": {"regex": "★ ([\\d,]+)", "type": "integer"}
    }
  }
}

Markdown Extraction Features

  • Numbered lists - 1. Item, 2. Item
  • Bullet lists - - Item, * Item
  • Regex patterns - Custom field extraction per item
  • Type coercion - integer, number, boolean
  • Required fields - Skip items missing required data
  • Multiline support - Capture descriptions across lines

Registry Configuration

registry.json

Registry Structure

{
  "schemas": {
    "FetchInput": {
      "type": "object",
      "properties": {
        "url": {"type": "string"},
        "max_length": {"type": "integer"}
      }
    }
  },
  "tools": [
    {
      "name": "fetch",
      "inputSchema": {"$ref": "#/schemas/FetchInput"},
      "server": {
        "command": "uvx",
        "args": ["mcp-server-fetch"],
        "transport": "stdio"
      }
    }
  ]
}

Tool Inheritance

{
  "tools": [
    {
      "name": "base_fetch",
      "server": {"command": "uvx", "args": ["mcp-server-fetch"]}
    },
    {
      "name": "research_url",
      "source": "base_fetch",
      "description": "Custom description"
    },
    {
      "name": "cached_fetch",
      "source": "research_url",
      "defaults": {"cache": true}
    }
  ]
}

Chain inheritance: cached_fetch → research_url → base_fetch

Key Data Structures

@dataclass(frozen=True)
class ServerConfig:
    command: str | None
    args: tuple[str, ...]
    url: str | None
    transport: str
    env: tuple[tuple[str, str], ...]  # Hashable!

@dataclass
class VirtualTool:

    description: str
    input_schema: dict
    server_id: str
    defaults: dict
    output_schema: dict | None
    text_extraction: dict | None  # Phase 2

MVA2: Constraint Hints

Planned - Not Yet Implemented

docs/tool-hints-plan.md

Capability Hints (Gateway-Enforced)

  • requiredPermissions - Remove if user lacks access
  • dataClassification - Track sensitive data flow
  • costTier - Remove based on billing plan
  • quotaKey - Remove if quota exhausted
  • networkDependency - Remove if offline
  • timingWindow - Remove outside hours

Binary decisions: tool is available or not

Behavior Hints (LLM Planning)

  • idempotent - Safe to retry
  • latency - "instant" | "seconds" | "minutes"
  • confirmationNeeded - Requires user approval
  • reversible - Can be undone
  • outputSize - Expected response size
  • reliability - Success rate for fallback planning

Soft guidance for intelligent planning

Request Flow


┌─────────────┐    ┌──────────────┐    ┌─────────────┐
│   Agent     │───►│  MCP Gateway │───►│ MCP Server  │
│   (LLM)     │    │              │    │  (Backend)  │
└─────────────┘    └──────────────┘    └─────────────┘
                         │
    1. Receive tool call │
    2. Look up VirtualTool
    3. Apply defaults
    4. Map to source tool
    5. Forward to backend
    6. Apply output projection
    7. Return transformed result
